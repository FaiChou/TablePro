# Query Parameters

Write `:name` placeholders in your SQL, fill values in an inline panel, and execute with native prepared statement binding. Parameter values persist across tab switches and app restarts.

## Usage

Write a query with `:name` placeholders:

```sql
SELECT *
FROM orders
WHERE customer_id = :customer_id
  AND status = :status
  AND created_at > :since;
```

Press `Cmd+Enter`. The parameter panel appears between the toolbar and the editor with fields for each detected parameter. Fill in values, then press `Cmd+Enter` again to execute.

The same flow works with **Execute All** (`Cmd+Shift+Enter`) for multi-statement scripts.

## Parameter Panel

Each parameter row has four controls:

| Control | Description |
|---------|-------------|
| **Name** | The parameter name from your query (read-only) |
| **Value** | Text field for the value |
| **Type** | String, Integer, Decimal, Date, or Boolean |
| **NULL** | Checkbox to bind NULL instead of the text value |

**Clear All** resets all value fields. Close the panel with the X button. The panel reappears automatically on the next execute if the query still contains parameters.

## How It Works

1. The editor scans for `:name` tokens outside string literals, comments, and PostgreSQL `::` casts
2. Detected names are shown in the panel. Existing values are preserved when the query changes: new parameters get empty fields, removed parameters disappear, matching names keep their values
3. On execute, `:name` placeholders are converted to the database's native style (`?` for MySQL/SQLite, `$1`/`$2` for PostgreSQL) and bound via prepared statements

<Tip>
Parameters use native prepared statement binding at the C library level, not string interpolation. This prevents SQL injection and handles quoting automatically.
</Tip>

## Supported Syntax

The scanner detects `:name` where `name` starts with a letter or underscore and contains letters, digits, or underscores.

| Pattern | Detected | Reason |
|---------|----------|--------|
| `:user_id` | Yes | Valid named parameter |
| `:status` | Yes | Valid named parameter |
| `':name'` | No | Inside a string literal |
| `-- :name` | No | Inside a line comment |
| `/* :name */` | No | Inside a block comment |
| `col::integer` | No | PostgreSQL type cast |
| `$$ :name $$` | No | Inside a dollar-quoted string |
| `:123` | No | Name must start with a letter or underscore |

Duplicate names (`:id = :id`) map to the same value field. Each occurrence binds the same value.

## Type Picker

The type determines how the value is passed to the database driver:

| Type | Binding behavior |
|------|-----------------|
| **String** | Bound as text (default) |
| **Integer** | Bound as text, validated by the driver |
| **Decimal** | Bound as text, validated by the driver |
| **Date** | Bound as text in your database's date format |
| **Boolean** | Bound as `true` or `false` |
| **NULL** | Binds NULL regardless of the value field |

## Persistence

Parameter values are saved with the tab:

- **Tab switching**: values are preserved when you switch between tabs
- **App restart**: values are restored from disk when the tab is reopened
- **Query history**: parameter values are recorded alongside the query

## Multi-Statement Queries

Each statement is scanned independently. A multi-statement script shares one parameter panel with all unique parameter names across all statements. On execute, each statement binds only the parameters it references.

```sql
INSERT INTO users (id, name) VALUES (:id, :name);
SELECT * FROM users WHERE id = :id;
```

Both statements share the `:id` parameter. The INSERT also uses `:name`.

## Safe Mode

Parameterized queries respect [Safe Mode](/features/safe-mode) settings. Dangerous queries (DROP, DELETE without WHERE, TRUNCATE) still require confirmation when safe mode is enabled.

## Settings

Toggle query parameters in **Settings** > **Editor** > **Query parameters (:name syntax)**. Enabled by default.

When disabled, `:name` tokens are sent to the database as-is (no detection, no panel, no conversion).

## Database Support

Query parameters work with all databases that support prepared statements:

| Database | Native binding |
|----------|---------------|
| MySQL / MariaDB | `?` placeholders via `mysql_stmt_bind_param` |
| PostgreSQL / Redshift | `$1`, `$2` via `PQexecParams` |
| SQLite / LibSQL | `?` placeholders via `sqlite3_bind_text` |
| DuckDB | `$1`, `$2` via prepared statements |
| Cloudflare D1 | `?` placeholders |
| ClickHouse | Client-side substitution |
| SQL Server | Client-side substitution |
| Others | Client-side substitution with proper escaping |
